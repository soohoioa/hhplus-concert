# Kafka 개념 학습 및 정리

## (이벤트 기반 시스템 확장 관점에서 이해하기)

------------------------------------------------------------------------

# 0. 이벤트 아이디어를 시스템 전체 관점으로 확장한다는 의미

예를 들어 "결제 성공"이라는 하나의 이벤트가 발생했다고 가정한다고 했을때,결제 성공 이후 시스템에서는 이런 일들이 연쇄적으로 일어날 수 있다.

-   예약 상태를 CONFIRMED 로 변경
-   좌석을 확정 처리
-   알림 발송 (문자/이메일)
-   통계/랭킹 반영
-   정산 집계 반영

이것을 모든 서비스를 서로 직접 API 호출로 연결하면 다음 문제가 생긴다.

-   서비스 간 결합도 증가
-   장애 전파 가능성 증가
-   기능 추가 시 기존 서비스 수정 필요
-   트래픽 폭증 시 전체 시스템 영향

그래서 등장하는 개념이 바로 **이벤트 기반 아키텍처(Event-Driven Architecture)** 이다.

결제 서비스는 단순히 -> "PaymentCompleted 이벤트"를 발행하고 다른 서비스들은 해당 이벤트를 구독하여 각자의 역할만 수행한다.

이 구조를 가능하게 해주는 대표적인 기술이 바로 Kafka 이다.

------------------------------------------------------------------------

# 1. Kafka는 무엇인가?

Kafka는 한 문장으로 정의하면

> 대규모 이벤트(메시지)를 빠르고 안정적으로 전달하고 저장하는 분산
> 이벤트 스트리밍 플랫폼

위 내용을 조금 더 쉽게 설명하면

-   Producer(생산자)가 이벤트를 Kafka에 보냅니다.
-   Kafka는 이벤트를 저장합니다. (디스크에 로그처럼 기록)
-   Consumer(소비자)가 이벤트를 읽어 처리합니다.

Kafka는 단순한 메시지 전달 시스템이 아니라, **이벤트를 로그 형태로 저장하는 분산 로그 시스템**이다.

즉, 메시지가 사라지는 것이 아니라 일정 기간 저장된다.

------------------------------------------------------------------------

# 2. Kafka를 사용하면 얻을 수 있는 장점과 단점

## 장점

### 1) 서비스 간 느슨한 결합

- 결제 서비스는 이벤트만 발행 
- 예약/알림/통계 서비스는 필요하면 구독
- 새 기능 추가 시 기존 코드 수정 최소화 가능


### 2) 높은 처리량

- Kafka는 대량 트래픽 처리에 매우 강함
- 초당 수만 ~ 수십만 이벤트 처리도 가능


### 3) 디스크 저장 + 재처리 가능

- 이벤트는 디스크에 저장
- 장애 발생으로 소비자가 못 읽어도 나중에 다시 읽어서 처리 (Replay 기능 가능)


### 4) 소비자 그룹(Consumer Group)을 통한 수평 확장

- 같은 토픽을 여러 인스턴스가 분산 처리 가능
- 인스턴스를 늘리면 처리량도 증가


### 5) 장애 격리

- 알림 서비스가 죽어도 결제는 계속 진행 가능 (이벤트는 Kafka에 남아있기 때문에 나중에 복구되면 이벤트 쌓인 것부터 처리)


## 단점

### 1) 운영 난이도

- 클러스터 구성, 모니터링, 파티션 설계, 리텐션, 장애 대응 등 고려할게 많음


### 2) 정확히 한 번(exactly-once) 처리 어려움

- 보통 기본 설계는 at-least-once(최소 1번 전달) 로 가는 경우가 많고, 그럼 중복 처리 가능성이 생겨서 소비자 측 idempotent(멱등) 처리가 중요해짐 (예: 결제완료 이벤트 중복으로 예약을 두 번 확정하면 안 됨)


### 3) 순서 보장 범위가 제한

- Kafka는 토픽 전체 순서가 아니라 파티션 내부 순서만 보장 -> 키 설계가 매우 중요


### 4) 최종 일관성(Eventual Consistency) 구조

- 이벤트 기반은 바로바로 동기 처리가 아니라, 각 서비스 처리 완료 시점이 다를 수 있음 -> 사용자 화면/상태 설계에서 이를 고려해야 함

# 3. Kafka의 특징과 주요 요소

Kafka를 이해하려면 다음 6가지를 알아야 한다.

## 1) Topic (토픽)

- 이벤트가 저장되는 논리적 채널
예: payment-events, reservation-events


## 2) Partition (파티션)

- 토픽을 여러 조각으로 나눈 것 (성능/확장 핵심)
- 파티션 단위로 병렬 처리가능 -> 파티션 내부에서만 순서 보장


## 3) Producer (프로듀서)

- Kafka에 이벤트를 보내는 쪽
예: 결제 서비스가 PaymentCompleted 이벤트 발행


## 4) Consumer (컨슈머)

- Kafka에서 이벤트를 읽는 쪽
예: 예약 서비스가 PaymentCompleted 이벤트 받아 예약 확정


## 5) Consumer Group (컨슈머 그룹)

- 같은 역할을 하는 컨슈머들의 묶음
- 그룹 내에서 파티션을 나눠서 처리
- 컨슈머 인스턴스를 늘리면 처리량이 늘어남


## 6) Broker / Cluster (브로커/클러스터)

- Kafka 서버 한 대 = Broker
- 여러 Broker가 모이면 Cluster를 이룸
- 장애 대비를 위해 이벤트는 복제되어 저장

# 4. Kafka의 핵심 기능

## 1) Pub/Sub (발행/구독) 구조

- 생산자는 이벤트만 던지고,
- 여러 서비스가 동시에 구독해서 처리 가능


## 2) 메시지 저장 + 리텐션 (보관 기간)

- 이벤트가 지나가면 가라지는게 아니라, 일정 기간 디스크에 저장됨
예: 3일치 결제 이벤트를 저장 같은 정책 가능


## 3) Offset (오프셋) 기반 읽기

- 컨슈머는 내가 어디까지 읽었는지 offset으로 관리
- 장애 발생 시 마지막 offset부터 재시작 가능
- 필요하면 과거 이벤트 재처리(replay)도 가능


## 4) 확장성(파티션 + 컨슈머 그룹)

- 파티션 수 증가 -> 병렬성 증가
- 컨슈머 인스턴스 증가 -> 처리량 증가
