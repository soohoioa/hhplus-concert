# 부하 테스트 계획서 및 수행 보고서
## (Hold API / Queue Gate 기반 콘서트 예매 시스템)

---
# 1. 부하 테스트 대상 선정 및 목적

## 1-1. 어떤 부분을 부하 테스트 해야 하는가?

### 대상 API
`POST /api/v1/reservations/hold` (Queue Gate 포함)

### 선정 이유

콘서트 예매 시스템에서 실제 트래픽이 폭주하는 지점은 **좌석 선점(hold)** 구간이다.

특히 본 시스템은 다음 구조를 포함하고 있다

- Redis 기반 대기열(Queue)
- Permit Scheduler 기반 ready 통과 구조
- DB 좌석 동시성 제어
- 좌석 상태 전이 (AVAILABLE → HELD → RESERVED)

이 구간은 다음 병목 가능성을 동시에 포함한다.

1. Redis ZSET 대기열 처리율
2. Permit 발급 처리율(batch-size / delay 정책)
3. DB 좌석 동시성 경합
4. HikariCP 커넥션 풀 제한
5. API 응답 지연(p95, tail latency)

즉, 단순 API 성능 테스트가 아니라 **동시성 제어 및 트래픽 폭주 대응 능력**을 검증하기 위한 핵심 구간이다.

---

## 1-2. 왜 이 부분을 부하 테스트 해야 하는가?

### 비즈니스 관점

콘서트 오픈 시점에는 짧은 시간에 수천~수만 명이 동시에 요청한다.

문제 발생 시
- 좌석 중복 예약
- 사용자 무한 대기
- API 타임아웃
- 서버 다운

이는 곧 서비스 신뢰도 하락과 매출 손실로 직결

따라서 사전 부하 테스트를 통해

- 최대 처리 가능 동시 요청 수 확인
- 병목 지점 사전 식별
- 적절한 배포 스펙 산정
- 설정 튜닝 방향 도출

이 반드시 필요

# 2. 부하 테스트 시나리오 설계

## 2-1. 테스트 흐름

실제 사용자 흐름을 기반으로 구성

1. `/queue/token` → 대기열 진입
2. `/queue/status` → ready 폴링
3. ready=true 시 `/reservations/hold` 호출

이 흐름을 그대로 k6 스크립트에 반영

---

## 2-2. 테스트 조건

- 최대 VU: 50
- ramping-vus: 10 → 30 → 50 → 0
- READY_TIMEOUT: 10초
- 성공 기준:
    - hold: 200 또는 409 (좌석 충돌은 정상 응답)
- 주요 관측 지표:
    - p95 응답시간
    - http_req_failed
    - ready timeout 발생 수
    - req/s
    - iteration 처리량
